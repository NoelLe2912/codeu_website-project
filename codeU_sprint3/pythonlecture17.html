Representing Playing Cards

Before learning inheritance, let's build a Card class. We'll use integer encoding to make cards easy to compare—mapping suits and ranks to numbers.

Encoding Suits and Ranks

Map suit names and ranks to integers:

Suits

0: Clubs

1: Diamonds

2: Hearts

3: Spades

Ranks

2-10: Face value

11: Jack

12: Queen

13: King

14: Ace

Class Variables

Store mappings as class variables (shared by all instances): 

class Card:
    \"\"\"Represents a standard playing card.\"\"\"
    
    suit_names = ['Clubs', 'Diamonds', 'Hearts', 'Spades']
    rank_names = [None, 'Ace', '2', '3', '4', '5', '6', '7',
                  '8', '9', '10', 'Jack', 'Queen', 'King', 'Ace']
    
    def __init__(self, suit, rank):
        self.suit = suit
        self.rank = rank

# Access class variables
Card.suit_names[0]
# 'Clubs'

Card.rank_names[11]
# 'Jack'

💡 Design Choice: rank_names starts with None so index 2 maps to '2', index 3 to '3', etc. Element 14 is 'Ace' (high).

Card String Representation

Use __str__ to create human-readable card representations. This makes testing and debugging much easier!

The __str__ Method

class Card:
    # ... class variables and __init__ ...
    
    def __str__(self):
        rank_name = Card.rank_names[self.rank]
        suit_name = Card.suit_names[self.suit]
        return f'{rank_name} of {suit_name}'

queen = Card(1, 12)  # Diamonds, Queen
print(queen)
# Queen of Diamonds

Comparing Cards

Define special methods to make comparison operators work with cards. This makes cards sortable and comparable!

Equality: __eq__

class Card:
    def __eq__(self, other):
        return self.suit == other.suit and self.rank == other.rank

queen = Card(1, 12)
queen2 = Card(1, 12)

queen == queen2
# True (same suit and rank)

six = Card(1, 6)
queen == six
# False

Less Than: __lt__

Compare by suit first, then rank: 

class Card:
    def to_tuple(self):
        return (self.suit, self.rank)
    
    def __lt__(self, other):
        return self.to_tuple() < other.to_tuple()

six < queen
# True (same suit, but 6 < 12)

Totally Ordered

With __eq__, __lt__, and __le__, cards are totally ordered:

• Any two cards can be compared

• Results are consistent (no contradictions)

• Cards can be sorted!

🎴 Tuple Comparison: Python compares tuples element by element, perfect for our suit-then-rank logic!

Creating a Deck

A Deck holds a collection of Card objects. We'll add methods to shuffle, sort, and deal cards.

Deck Class

class Deck:
    def __init__(self, cards):
        self.cards = cards
    
    @staticmethod
    def make_cards():
        cards = []
        for suit in range(4):
            for rank in range(2, 15):
                card = Card(suit, rank)
                cards.append(card)
        return cards

# Create a full deck
cards = Deck.make_cards()
deck = Deck(cards)
len(deck.cards)
# 52

Deck Operations

import random

class Deck:
    def take_card(self):
        return self.cards.pop()
    
    def put_card(self, card):
        self.cards.append(card)
    
    def shuffle(self):
        random.shuffle(self.cards)
    
    def sort(self):
        self.cards.sort()  # Uses Card.__lt__

# Deal a card
card = deck.take_card()

# Shuffle
deck.shuffle()

# Sort
deck.sort()

🔄 Delegation: Deck.sort just calls list.sort, passing responsibility to the built-in method. This is called delegation!

Inheritance: Parents and Children

Inheritance lets you create a new class based on an existing one. The new class (child) inherits all methods and attributes from the parent, but can add or override them.

Defining a Child Class

Put the parent class name in parentheses: 

class Hand(Deck):
    \"\"\"Represents a hand of playing cards.\"\"\"
    
    def __init__(self, label=''):
        self.label = label
        self.cards = []

# Hand inherits from Deck
hand = Hand('player 1')
hand.label
# 'player 1'

# Can use Deck methods!
card = deck.take_card()
hand.put_card(card)  # Inherited from Deck

Inheritance Rules

1. Child instances have all parent attributes (plus extras)

2. Child has all parent methods (plus extras)

3. Child methods can override parent methods

4. Overriding methods should be compatible (same parameters, compatible results)

Parent Class (Deck)

• General purpose

• Basic operations

• Foundation for children

Child Class (Hand)

• Specialized purpose

• Inherits parent operations

• Adds specific features

Polymorphism

Methods that work with multiple types are called polymorphic. This makes code more flexible and reusable!

Polymorphic Method Example

class Deck:
    def move_cards(self, other, num):
        for i in range(num):
            card = self.take_card()
            other.put_card(card)

# Works with any combination!
deck.move_cards(hand, 5)      # Deck → Hand
hand.move_cards(deck, 2)      # Hand → Deck
hand1.move_cards(hand2, 3)    # Hand → Hand

Why Polymorphism Works

The method works because:

• Both Deck and Hand have take_card() and put_card()

• The method doesn't care about the specific type

• It just needs objects with the right methods

🎭 Polymorphism: "Many forms" — one method works with many types. This is a key benefit of inheritance!

Specialization

Create specialized versions of classes for specific purposes. BridgeHand inherits from Hand but adds bridge-specific functionality.

BridgeHand Class

class BridgeHand(Hand):
    \"\"\"Represents a bridge hand.\"\"\"
    
    hcp_dict = {
        'Ace': 4,
        'King': 3,
        'Queen': 2,
        'Jack': 1,
    }
    
    def high_card_point_count(self):
        count = 0
        for card in self.cards:
            rank_name = Card.rank_names[card.rank]
            count += BridgeHand.hcp_dict.get(rank_name, 0)
        return count

hand = BridgeHand('player 2')
deck.move_cards(hand, 5)
hand.high_card_point_count()
# Returns point total

Specialization Benefits

• Reuses Hand functionality (take_card, put_card, etc.)

• Adds bridge-specific features (scoring)

• No need to modify the parent class

• Natural reflection of problem structure

🎯 Design Pattern: Specialization creates domain-specific classes from general-purpose ones!

The Liskov Substitution Principle

Named after computer scientist Barbara Liskov, this principle ensures inheritance is used correctly.

The Principle

Any code designed for a parent class should work with child classes:

✅ Child instances: Have all parent attributes (can have more)

✅ Child methods: Include all parent methods (can have more)

✅ Overriding: New methods take same parameters, return compatible results

Breaking the Rules

If you violate these rules:

• Code expecting a parent won't work with children

• Your program will collapse like a house of cards!

• Debugging becomes very difficult

📜 Design Law: Follow the Liskov Substitution Principle to create robust, maintainable inheritance hierarchies!

Debugging Inheritance

Inheritance can make code harder to read because method definitions are spread across classes. Use these techniques to track execution flow.

Finding Method Definitions

def find_defining_class(obj, method_name):
    \"\"\"Find the class where a method is defined.\"\"\"
    for typ in type(obj).mro():
        if method_name in vars(typ):
            return typ
    return f'Method {method_name} not found.'

hand = BridgeHand('player 3')
find_defining_class(hand, 'shuffle')
#

Method Resolution Order (MRO)

Python searches for methods in this order:

1. The instance's class (BridgeHand)

2. The parent class (Hand)

3. The grandparent class (Deck)

4. And so on up the hierarchy

🔍 Debugging Tip: Add print statements at method beginnings to trace execution flow through your inheritance hierarchy!
